# 소담(SODAM) 다중 페르소나 평가·토론 종합 보고서

> 작성: 메인 중재자(전권 위임) · 일시: 2026-06-20
> 방법: 6개 페르소나 독립평가(R1) → 메인 코드 직접검증 → 2개 진영 전략토론(R2) → 메인 적격판정·결정
> 본 보고서의 모든 코드 결함은 **메인이 레포에서 직접 확인**한 사실이며, 검증 없는 페르소나 주장은 채택하지 않았다.

## Part 0. 한 페이지 요약 (Executive Summary)

**판정: 조건부 NO-GO → 3~5일 사전수정 게이트 통과 시 베타 GO.**

이번 평가의 가장 중요한 산출물은 기능 목록이 아니라 **기존 "코드 차단 0 / 보안 수리 완료" 서사가 코드 수준에서 사실이 아님을 검증**한 것이다. 도메인 전문 페르소나(노무·플랫폼)가 지목하고 메인이 직접 코드로 확인한 **출시 차단급 P0 3건**이 실재한다.

| # | 검증된 P0 | 상태 | 한 줄 |
|---|---|---|---|
| P0-1 | **NFC 검증 = 스텁** | ✅ 코드 확인 | `'SODAM-'`+10자 형식만 통과, 매장-태그 대조 없음 → 임의 문자열로 대리출근 |
| P0-2 | **GPS 좌표 평문 로깅** | ✅ 코드 확인 | `ServiceLoggingAspect`가 모든 서비스 인자를 INFO 평문 → 출근 위경도·employeeId 노출(위치정보법) |
| P0-3 | **급여/시급 RBAC 부재** | ✅ 코드 확인 | Wage/Payroll/Master 컨트롤러 `@PreAuthorize` 0건, `anyRequest().authenticated()`뿐 → 인증된 직원이 급여 엔드포인트 호출 |

**핵심 통찰(양 진영 합의):** P0-1과 P0-3의 결합은 *대리출근 → 급여 조작* 2단계 사기 체인을 만든다. 더 중요한 건 — **이 구멍이 열린 채 베타를 돌리면 거기서 나온 activation·리텐션 데이터 자체가 오염되어 그로스 판단도, 투자자 매출 검증도 무의미해진다.** 따라서 P0 수정은 "성장보다 먼저"가 아니라 "성장을 측정하기 위한 전제"다.

**범위 규율(성장 진영 합의):** 단, P0 수정은 골드플레이팅 없이 **3~5일로 바운드**하고, 코드를 못 만지는 **외부 행정 게이트(토스 가맹 심사·사업자등록·법무 검토)는 지금 당장 병렬로** 시작한다. 노무 P1·전면 RBAC 매트릭스·전체 관측성은 베타 중 트랙으로 내린다.

**전략 재설정(투자자):** M12 매출 목표를 환급·세무 채널 포함 ₩2.34M에서 **구독 단독 floor ₩884K(유료 ~52~65개)** 로 재설정. activation 실측 전에는 펀딩이 아니라 부트스트랩. 1인 SPOF가 구조적 최대 리스크.

## Part 2. 메인 코드 검증 결과 (이 보고서의 사실 기반)

### 2.1 사실로 확인 — 출시 차단급
- **P0-1 NFC 스텁** — `backend/.../service/NfcVerificationService.java`. 클래스 주석 "NFC 태그 검증 서비스(스텁 구현)". `verifyTag(storeId, tagId)`는 `tagId.startsWith("SODAM-")` + `length>=10`만 검사, `storeId`는 받기만 하고 미사용("TODO: 정책 확정 시 매장별 등록/만료/블랙리스트"). **매장에 실제 부착된 태그와의 대조가 없다.**
- **P0-2 GPS 평문 로깅** — `backend/.../aop/logging/ServiceLoggingAspect.java:30` `log.info("실행 시작: {}.{}() 매개변수: {}", className, methodName, Arrays.toString(args))`. 출퇴근 서비스 호출 시 위도·경도·employeeId가 INFO 평문으로 적재.
- **P0-3 RBAC 부재** — `WageController/PayrollController/MasterController` `@PreAuthorize`/`hasRole` **각 0건**. `SecurityConfig`는 `@EnableMethodSecurity` 켜져 있으나 인가 규칙은 `permitAll`(일부 공개경로) + `anyRequest().authenticated()`뿐. `@PreAuthorize`는 전체 5건으로 일부 컨트롤러(IDOR 수리분 추정)에만 존재. **급여·시급·마스터 엔드포인트는 역할 게이트가 없다.**

### 2.2 과장으로 판명 — 채택하지 않음
- **노무 §48 "공제총액 불일치"** → **버그 아님.** `PayrollService.addDeductionRows()`는 4대보험 정책 시 4개 항목을 개별 표기하고 `공제총액 = payroll.getTaxAmount()`. `calculateTax(FOUR_INSURANCES)`는 `socialInsuranceCalculator.totalEmployeeDeduction()` = 4대보험 합계를 반환하므로 **공제총액 == 항목 합계로 일치**한다. (단 4대보험 직원 근로소득세 간이세액 미공제는 별개 — 저임금 파트타이머는 대개 면세구간이라 P2로 방어 가능. 고임금 직원 보유 매장에선 보완 권장.)

### 2.3 코드 인용은 했으나 미심층검증 (페르소나 신뢰, P1로 유지)
- 주휴 소정근로일 미설정 폴백 → 결근자 주휴 과지급 가능(노무 R-1)
- 국민연금 기준소득월액 상한 2026.07 미갱신 자동화 부재(노무 R-2)
- §17 임금 지급방법(계좌이체/현금)·구성항목 분해 필드 부재(노무 R-4)
- 오프라인 큐 `queuedAt` 미전송 → flush 시 서버시각 기록으로 출근시각 왜곡(플랫폼 R-4)

## Part 3. 쟁점별 토론 적격판정 (메인 ruling)

### 쟁점 A — 출시 게이트: "코드는 끝났다" vs "코드 P0 실재"
- **PO 주장:** 코드차단 0, 노트북 닫고 행정 뛰어라.
- **플랫폼·노무 주장:** 검증 P0 3건은 코드 차단.
- **🟢 메인 판정 — 플랫폼·노무 승.** PO의 전제("코드차단 0")는 검증으로 반증됐다. 단 PO의 *외부 게이트가 진짜 임계경로*라는 지적은 유효하며 R2에서 양 진영이 수용. **결론: 코드 P0 3건 = 베타 전 필수, 외부 행정 = 즉시 병렬.**

### 쟁점 B — P0를 베타 전이냐 베타 중이냐
- **성장 진영:** 클로즈드 베타는 위협 통제됨 → NFC·전면RBAC는 베타 중, GPS 마스킹만 베타 전.
- **De-risk 진영:** 베타 사용자가 실제 급여를 정산하는 순간 사기 체인이 재현가능해지고, **그 환경의 activation 데이터는 오염**된다.
- **🟢 메인 판정 — De-risk 승(결정적 논거: 데이터 오염).** 단 성장 진영의 *범위 규율*은 채택: 3~5일 바운드, GPS-우선 aha 경로는 UX로 유지. **NFC 대조는 스텁이 이미 `storeId`를 받으므로 Store-태그 조회만 붙이면 1~2일 — 미룰 이유 없음.** RBAC도 비즈로직 변경 없는 스캐폴딩 2일. → **3건 모두 베타 전.**

### 쟁점 C — activation vs 수익화 게이팅
- **그로스:** 출퇴근→급여 핵심 루프는 FREE로 보호, 게이팅은 가치경험 이후(멀티매장·PDF·집계)에.
- **수익화:** 게이팅 전수부착으로 ARPU 방어, 직원상한·PDF 무료는 전환 죽인다.
- **🟢 메인 판정 — 분할 채택.** ①핵심 루프(첫 출퇴근→첫 급여 미리보기 *숫자*)는 FREE 보호(그로스 승) ②발급·집계·멀티매장은 게이팅 유지(수익화 승) ③**직원상한 2→3 A/B**와 **월 1회 무료 명세서 실발급 후 페이월** A/B는 양측 공통 합의 → 채택. ④게이팅 전수부착은 출시 전 필수(수익화 R-1, ARPU 직결).

### 쟁점 D — 매출 목표 현실성
- **🟢 메인 판정 — 투자자 승.** M12 목표를 **구독 단독 floor ₩884K(유료 52~65개)** 로 재설정, 세무·환급·노무 채널은 업사이드로만 보고(법적 게이트·유권해석 의존). 현실 ₩2.34M은 레버 3개 동시성공(추정 합산확률 ~22%) 전제라 계획 기준선으로 부적합.

## Part 7. 메인의 최종 결론

1. **"출시 준비 완료" 서사를 정정한다.** 코드 차단은 0이 아니라 **검증된 P0 3건**이 남아 있었다. 이 발견 자체가 이번 토론의 최대 성과다.
2. **즉시 실행:** TIER 0 6개 항목(3~5일) + 외부 행정 병렬 착수. 이 중 GPS 마스킹·게이팅 부착·이벤트 로깅·카피 교체는 **메인 자율 실행 가능**, NFC 대조·§17 필드는 Flyway 작성까지 자율·실행 승인.
3. **목표 재설정:** 구독 단독 floor ₩884K. activation 실측이 모든 전략의 선행조건.

# Part 8. 라운드3 부록 — 코드 정밀학습 토론 (2026-06-21)

> 6개 페르소나가 **각자 담당 코드를 먼저 정밀 학습**한 뒤 토론. 이전 2라운드가 놓친 코드 근거 신규 발견 다수 + **메인 R2 판정의 정정**. 아래 보안 항목은 메인이 직접 코드로 재확인했다.

## 8.1 ⚠️ 메인 R2 판정 정정 — "RBAC 전무"는 과장
- **정정:** Part 0/2의 P0-3 "급여 RBAC 부재(전무)"는 **과장**이었다. 커스텀 RBAC 레이어가 실재한다: `@MasterOnly`·`@EmployeeOrMaster` 어노테이션 + `StoreAccessGuard` 서비스가 Wage/Payroll/Master 컨트롤러에 **광범위 적용**돼 있다. (예: `WageController` 쓰기 4메서드 `@MasterOnly`, `MasterController` 클래스 `@MasterOnly`.) R2 grep이 Spring `@PreAuthorize`/`hasRole`만 찾아 커스텀 레이어를 놓쳤다.
- **그러나 더 정확하고 더 위험한 진짜 구멍을 코드로 확인:** 권한(역할)은 막혀 있으나 **신원 바인딩(IDOR)이 누락**된 엔드포인트가 있다. ↓

## 8.2 ✅ 신규 검증 P0 — IDOR (메인 코드 확인)
| ID | 위치 | 결함 | 악용 | 소요 |
|---|---|---|---|---|
| **P0-A 출퇴근 IDOR** | `AttendanceController.checkIn/checkOut` | `request.getEmployeeId()`를 principal과 대조 안 함, guard 미호출 | 인증된 사용자가 **임의 employeeId로 대리출근/타인 명의 출근** | 0.5일 |
| **P0-B 급여 IDOR** | `PayrollController /{payrollId}/details·/pdf` | `@MasterOnly` 없음 + guard 미호출 | **임의 payrollId로 타인 급여명세서·PDF 열람**(임금정보 PII) | 0.5일 |

> **재프레이밍:** P0-A(출퇴근 IDOR)는 **NFC 스텁(P0-1)보다 치명적**이다. NFC를 완벽히 막아도 이 REST 경로로 출근 조작이 가능하기 때문. 따라서 fraud 차단의 1순위는 NFC가 아니라 **출퇴근 엔드포인트의 `guard.assertSelf(principal, employeeId)` 추가**다. (`StoreAccessGuard`에 `assertSelf`/`assertCanViewEmployee` 이미 구현됨 — 호출만 추가.)
> P0-1 NFC: GPS 전용 출시로 단기 우회 가능. 단 `/verify/nfc` 스텁 엔드포인트는 닫고, 매장-태그 DB 대조는 베타 중 완성.
> P0-2 GPS 로깅 마스킹: 불변(유효). `ServiceLoggingAspect:30` `Arrays.toString(args)` → 위경도 마스킹.
